# ADR-0007: Pin Zig 0.16.0 for this zentinel version

**Status:** Accepted
**Date:** 2026-05-20

## Context

ADR-0001 selected a moving latest-stable Zig policy to avoid supporting multiple compiler versions. The repository later made task `005` depend on verifying the current official latest stable release, which created an avoidable future blocker for autonomous agents.

The direct product decision for this zentinel version is now to pin one concrete compiler version. The local development toolchain reports Zig `0.16.0`, and task `005` can implement deterministic validation against that documented pin without performing a live release lookup.

## Decision

zentinel supports exactly Zig `0.16.0` for this zentinel version. The compiled-in supported version must be stored in one version-policy module and compared against `zig version`.

Other stable Zig versions and nightly builds are unsupported unless a future ADR and task update the pin. The task system must not require live latest-stable discovery to begin version-policy implementation.

## Alternatives Considered

- **Keep a moving latest-stable policy.** Rejected because it requires live release-source access or an offline mirror contract before task `005` can proceed.
- **Support a version range.** Rejected because it reintroduces parser, diagnostics, fixture, and compiler-internal ambiguity.
- **Use whatever Zig is installed.** Rejected because it makes report evidence and cache keys depend on machine-local state.

## Consequences

**Positive.** Future agents can implement the version checker with deterministic tests and no network prerequisite. Reports, cache keys, fixtures, and CI can name the exact compiler version.

**Negative.** The repository must update this ADR, `docs/ZIG_VERSION_POLICY.md`, task requirements, CI, and user-facing docs when zentinel intentionally moves to a newer Zig release.
